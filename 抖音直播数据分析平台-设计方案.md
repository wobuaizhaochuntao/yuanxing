# 抖音直播数据分析平台 — 产品设计方案

> 目标：从概览到细节，5 秒内判断一场直播好坏，3 步内定位问题根因  
> 原则：少看板、多诊断；先给结论，再给数据支撑

---

## 一、整体架构（三层漏斗）

```
┌─────────────────────────────────────────────────────────────┐
│  L1 概览层：达人看板                                          │
│  —— 今天谁播得好？谁出了问题？                                   │
├─────────────────────────────────────────────────────────────┤
│  L2 场次层：直播列表 + 单场卡片                                  │
│  —— 这场直播整体打几分？哪里及格哪里不及格？                      │
├─────────────────────────────────────────────────────────────┤
│  L3 细节层：单场深度复盘（5 个模块）                              │
│  —— 什么时间点出了什么问题？怎么改？                               │
└─────────────────────────────────────────────────────────────┘
```

---

### 3.3 直播综合评分模型

```
综合评分 = 流量得分 × 0.25 + 转化得分 × 0.30 + 商品得分 × 0.25 + 留存得分 × 0.20

各维度得分（百分制，与近7场均值对比）：
- 流量得分：场观、进房人数、在线峰值
- 转化得分：商品点击率、点击成交率、ROI、GPM
- 商品得分：商品点击表现、GMV贡献集中度、退款率
- 留存得分：人均停留时长、互动率、粉丝占比
```

### 3.4 关键交互
- 点击卡片 → 进入单场深度复盘（L3）
- 卡片hover → 显示更详细的异常指标列表
- 支持按评分/GMV/时间排序

## 四、L3 单场深度复盘层（核心）

单场复盘是整个平台的核心，采用 **"1+4" 结构**：
- **1 个实时总览**：开播中/已结束直播的核心看板
- **4 个深度模块**：漏斗分析、分钟级复盘、商品分析、人群洞察

### 4.0 导航结构

```
左侧导航栏（固定）：
├── 实时总览（默认页）
├── 成交漏斗
├── 分钟级复盘
├── 商品分析
├── 人群洞察
├── 评论洞察
└── 直播诊断（自动生成报告）
```

## 五、自动诊断与归因系统（核心引擎）

### 5.1 诊断规则引擎

基于你现有的数据能力，设计一套**规则+阈值**的诊断引擎，自动生成结论：

```
┌─────────────────────────────────────────────────────────────┐
│  【诊断维度】                                                  │
│  1. 流量诊断：场观、进房人数、在线人数、流量来源结构             │
│  2. 承接诊断：人均停留时长、互动率、进房→停留转化率              │
│  3. 商品诊断：商品点击率、点击成交率、GPM、退款率                 │
│  4. 转化诊断：下单率、支付率、成交率、客单价                      │
│  5. 内容诊断：评论情感分析、高频问题、负面反馈占比                │
│                                                             │
│  【诊断规则示例】                                              │
│  ┌──────────────────────────────────────────────────────┐    │
│  │  IF 商品点击率 < 近7场均值 × 0.8                      │    │
│  │  THEN 标记：商品点击率偏低                              │    │

### 5.2 归因逻辑

```
一场直播GMV不理想，归因优先级：

第一层：流量问题？
  → 场观是否足够？（如果场观<均值50%，先解决流量）
  → 进房率是否正常？（如果进房率<均值，可能是封面/标题问题）

第二层：承接问题？
  → 人均停留时长是否正常？（如果<均值，内容/节奏有问题）
  → 在线人数走势是否健康？（如果持续下滑，可能是节奏问题）

第三层：商品问题？
  → 商品点击率是否正常？（如果<均值，商品展示/讲解有问题）
  → 点击成交率是否正常？（如果<均值，价格/信任/详情页有问题）
  → 退款率是否偏高？（如果>均值，商品质量/描述有问题）

第四层：转化问题？
  → 下单率/支付率是否正常？（如果<均值，临门一脚有问题）
  → 客单价是否正常？（如果<均值，商品组合/升单话术有问题）
```

```

## 八、交互设计要点

### 8.1 对比维度
- 全局支持"较上一场"和"近7场均值"两种对比
- 对比结果用颜色+箭头直观展示（红绿/上下箭头）

- 实时总览：1分钟（开播中）/ 5分钟（已结束）
- 分钟级复盘：5分钟（默认）/ 10分钟 / 15分钟可选

### 8.3 事件标注
- 分钟级图表支持标注主播动作（上架商品、福袋、推流、切换主播等）

### 8.4 下钻路径

```
达人概览 → 点击达人 → 场次列表 → 点击场次 → 实时总览

                                            ↓

```

## 九、设计原则总结

| 原则 | 说明 |
|------|------|
| **结论先行** | 先告诉用户"这场直播打几分、哪里有问题"，再给数据 |
| **三层下钻** | 概览→场次→细节，每层只展示该层需要的信息 |
| **颜色即语义** | 红/绿/黄统一含义，不混淆用户 |
| **自动诊断** | 减少人工分析，让系统生成结论和建议 |
| **可对比** | 所有数据默认与上一场/近7场对比，没有对比的数据无意义 |
| **事件归因** | 分钟级数据必须关联主播动作，才能归因 |

---

> 这份设计遵循"简单但不简陋"的原则，核心功能聚焦在**快速诊断**和**自动归因**上，不做复杂的多维分析，让运营每天花10分钟就能知道昨天/今天的直播哪里好、哪里差、怎么改。
